Description

The VersionOne Quality Center Integration (V1QualityCenter) allows an organization to leverage its investment in Quality Center as part of an integrated platform for agile development. This integration affords agile teams the ability to:

  • Create Tests in Quality Center from Tests defined in VersionOne
  • Update the status of a VersionOne Test from the corresponding Quality Center Test
  • Create Defects in VersionOne from Defects in Quality Center
  • Mark Quality Center Defects as Fixed when the corresponding VersionOne Defect is closed

System Requirements

VersionOne:

  • 8.3 and above, including Team Edition

Integration Server Operating System:

  • Windows 2003, 2008 with .NET Framework 4.0

Quality Center:

  • Tested With Release 9.1.0.4353, 9.2.0.4862, 10.0, 11.0

Download

The latest version of V1QualityCenter is available from the VersionOne Community site.

Installation

These installation instructions assume that Quality Center is already installed, configured, and working properly.

  1. Determine Install Location

    V1QualityCenter can be installed on any server with network access to both VersionOne and Quality Center. Exact placement should be determined by your internal software management requirements. The integration server must meet the System Requirements stated above.

  2. Extract Files

    Download V1QualityCenter using the link above and extract it into a folder of your choice.

  3. Configure

    Configuration for the Quality Center Integration is a 3 step process.

    1. Configure Quality Center
    2. Configure VersionOne
    3. Configure ServiceHost
  4. Start integration

    Open up the command prompt, navigate to your installation folder, and run the following command:
    VersionOne.ServiceHost.exe

    If you have configured your system properly, you should see several [Info] messages followed by a [Startup] message.

  5. Test the integration

    Details on testing the integration are available below in the testing section.

  6. Install as a Windows Service

    Run the following command from the console window:
    VersionOne.ServiceHost.exe --install

    This command installs the service so it will run under the account NT AUTHORITY\Local Service. Local Service must be given access privileges to the directory where the V1QualityCenter was installed so it can store its state and write to log files. Follow the steps below to change the security on the installation directory:

    1. Right click the installation folder from Windows Explorer.
    2. Select "properties".
    3. Select the "Security" tab.
    4. Click the "Add" button.
    5. Enter "Local Service" and click "OK".
    6. Click the "Allow" checkbox for the "Full Control" row.
    7. Click "OK" to save the changes.

Configuration

Configure Quality Center

This section describes how to configure Quality Center for use with VersionOne.

  1. Create a User Field on "Test" for storing the VersionOne identifier (Tools | Customize | Project Entities).

  2. Select or create a user in Quality Center to represent the VersionOne system.

    This user must have sufficient rights to accept and modify defects. Additionally, this user must have access to all Quality Center projects that contain defects that will be pushed to VersionOne.

Configure VersionOne

  1. Add "Quality Center" to the list as a valid "Source".

    Skip this step if you are configuring a VersionOne Team Edition instance.

  2. Select or create a user in VersionOne to represent the Quality Center system.

    This user must have sufficient rights to create Defects, and accept and modify Tests in VersionOne. Additionally, this user must have access to all VersionOne projects that contain Tests and Defect statuses that will be pushed to Quality Center. The default filters expect this user's "Short Name" (i.e. Owners Nickname) to be "qc".

  3. Determine where to store the Quality Center identifier in VersionOne.

    The integration needs a text field in VersionOne to store the Quality Center identifier, which is a reference to the domain, project, and defect ID in Quality Center. By default this is the "Reference" field. If you are already using this field for a different purpose, you will need to create a custom text field and note the name.

Configure the Integration

To configure the Quality Center integration, run the ServiceHost configuration tool
ServiceHostConfigTool.exe

The following section describes how to configure the VersionOne Quality Center Integration using the configuration tool. This tool updates the VersionOne.ServiceHost.exe.config file located in the same directory.

  1. On the "General" tab specify your VersionOne connection details.

    The following table describes the fields on this tab:

    Field Description
    Server URL This is the URL to your VersionOne server
    Username VersionOne user that will create defects (i.e. the user created in the previous steps)
    Password Password for the specified user
    Use Windows Integrated Authentication Check this box if your VersionOne server is configured to use Windows Integrated Authentication

    If there is a proxy between this machine and the VersionOne instance, you'll also need to configure the following settings:

    Field Description
    Use Proxy For Connection Determines if the integration tries to connect through a Proxy
    Proxy URL This is the URL to your Proxy Server
    Proxy Username The username that will get you past this proxy
    Proxy Password The password for the Proxy Username
    Proxy Domain Name of Proxy Domain
  2. Once the VersionOne parameters are specified, press Verify button to continue.
  3. On the "Defects" tab specify the VersionOne field that will hold the Quality Center ID. This is typically the "Reference" field.

    The following table describes the fields on this tab:

    Field Description
    Reference Field Name Defect field used to hold Quality Center ID. By default this is the Reference field.
    Disabled Check this box to disable polling VersionOne for Defect updates. If disabled, defects will not be transferred to VersionOne from Quality Center nor will defects closed in VersionOne send updates to Quality Center.

    You must restart the service after making a config file change.

  4. On the "Test Services" tab specify how VersionOne tests will be pushed and updated to/from Quality Center.

    The following table describes the fields on this tab:

    Field Description
    Disabled Check this box to disable polling VersionOne for Test updates.
    Reference Field Name Field on a VersionOne Test used to hold Quality Center ID. By default this is the Reference field.
    Passed Test Status The Test Status value in VersionOne that indicates a test succeeded.
    Failed Test Status The Test Status value in VersionOne that indicates a test failed.
    Change Comment The change comment put into history for any modifications the service makes in VersionOne
    Description Suffix A suffix for the description of any defects that the service creates; Embedded Rich Text (HTML) is valid in this suffix
    Create Defect Indicates whether the TestWriterService should create defects when a failed test is closed in VersionOne. Valid values are:
    * All: The integration will create a VersionOne Defect for any closed test in the project when the test fails.
    * CurrentIteration: The integration will only create VersionOne Defects for closed tests within the current iteration when the test fails.
    * None: The integration never create VersionOne defects when a test fails.
    Base Query Filter The VersionOne query used to locate Tests in VersionOne that need to be created in Quality Center. The default query will find Tests that have a blank Reference and the Test Owner Short Name (i.e. Owner's Nickname) is "qc" and Status value is blank.
    Project Mapping Specify the list of projects in VersionOne that should be queried for test cases to send to Quality Center. Multiple VersionOne projects can be specified. If the project has children, you can include those children when querying with the "Incl. Child Projects" checkbox. The "Quality Center Project ID" is not the actual name of the project in Quality Center. Instead, the ID specified here is simply a reference to the corresponding "Project ID" in the project mappings within the "Quality Center" tab on the config tool. The actual Quality Center project specification is created on the "Quality Center" -> "Project and Priority Mappings" tab.

    You must restart the service after making a config file change.

  5. On the "Quality Center" tab, specify your Quality Center connection details and the data transfer behavior.

    Field Description
    Disabled Check this box if you want to disable polling Quality Center for updates to tests and new defects.
    URL Fully qualified URL to your Quality Center instance
    Username Valid Quality Center user
    Password Password for specified user
    1. Click Verify to ensure the connection parameters are correct.
    2. Configure the Defect Search Criteria.
      Field Description
      Field Name Quality Center Field to include in query for Defects you want created in VersionOne
      Field Value Value expected in Field to trigger defect creation in VersionOne

      The typical rows specified are BG_RESPONSIBLE with the Quality Center user name representing the VersionOne user, and BG_STATUS with the status value to trigger defect creation in VersionOne, such as "New".

    3. Specify the VersionOne Source value (the value you created under step 1 of Configure VersionOne).

      VersionOne Team Edition contains a Source value of "External System"

    4. Specify the Quality Center Defect status value to set back in Quality Center after the Defect is created in VersionOne.
    5. Specify the Quality Center Defect status value to set back in Quality Center after the Defect is closed in VersionOne.
  6. Configure which Quality Center Projects are polled (Project and Priority Mappings)

    Project Mapping allows you to specify which Quality Center Projects are polled and where to create Defects in VersionOne.

    Field Description
    Project ID The integration identifier for this project (the same value you entered as an identifier on the "Test Service" tab)
    Domain Quality Center Domain
    Project Quality Center Project Name
    Folder Name of Quality Center folder to use for creating tests (if it doesn't exist, it will be created)
    VersionOne ID Field Quality Center custom field that will hold the VersionOne Test ID (the one you created under "Project Entities" in Quality Center)
    VersionOne Project VersionOne Project to use for creating Defects

    You can map multiple Quality Center projects to the same VersionOne project.

  7. Map your Priority Values.

    Priority mapping allows you to configure how the VersionOne Defect priority value is set based on the Quality Center Priority value. The algorithm for mapping is simple, if the Quality Center Priority value is in the mapping, the VersionOne defect value is set to the mapped value. If the Quality Center Priority value is not found, the VersionOne defect value is not set.

    To add a Priority mapping you need to do the following:

    1. Click on the "Project and Priority Mappings" tab.
    2. In the Priority Mappings grid, select a VersionOne Priority value from the dropdown.
    3. Supply the corresponding Quality Center Project value name.

    To remove a mapping:

    1. Select the desired row.
    2. Click the Delete current row button.
  8. Save your changes and exit the program.

Testing

Each section below describes how to test one aspect of the VersionOne Quality Center Integration. To shut down the VersionOne Quality Center Integration after testing, press "Q" in the console window.

To Create a Quality Center Test from a VersionOne Test

  1. In VersionOne
    1. Select a project you defined on the "Test Service" tab in the configuration tool (Configure ServiceHost).
    2. Select a Test and set the field to match the criteria defined on the "Test Service" tab. If you used the default configuration, you need to set the owner to the "qc" user and ensure that the test has no value in the Status or Reference fields.
  2. In Quality Center
    1. Connect to the project you expected the test to appear.

    2. Verify that the test appears in the Test Plan section under the folder configured on the "Quality Center" tab in the Project section.

To Update the Status of a VersionOne Test from Quality Center

  1. In Quality Center
    1. Assign the test created from VersionOne to a TestSet.
    2. Change the status of this test.
  2. In VersionOne
    1. Verify the test status has changed.

To Create a VersionOne Defect from Quality Center

  1. In Quality Center
    1. Assign a Defect to the "Version One" user.
    2. Set the Defect State to "New".
    3. Verify the defect state has changed to Open.
  2. In VersionOne
    1. Verify the defect is in the backlog for the VersionOne project specified on the "Quality Center" tab in the configuration tool

To Update a Quality Center Defect from VersionOne

  1. In VersionOne
    1. Close a Defect that was created from Quality Center.

  2. In Quality Center
    1. Verify the defect status is set to "Fixed".
